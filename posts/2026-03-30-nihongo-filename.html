HTMLファイルを日本語ファイル名にしたら、地味に不具合が出続けた話

2026年3月26日
失敗学び

ブラウザのアドレスバーに並んだ文字列を見て、固まった。

https://example.com/%E3%82%A2%E3%83%8B%E3%83%A1.html

これが何を意味しているか、わかるだろうか。僕のサイトの「アニメ.html」ページのURLだ。日本語のファイル名がURLエンコードされると、こうなる。人間が読める文字が、パーセント記号と16進数の羅列に変わる。美しいと思って名づけたファイル名が、URLの中では暗号になっていた。

「日本語の方がわかりやすいじゃん」

事の始まりは、個人サイトのファイル構成を決めたときだった。HTMLファイルをフォルダに並べていく。トップページはindex.html。ここは英語だ。さすがにindex.htmlを変える人はいない。

問題は、他のページだった。自分の趣味を紹介するページ。アニメの感想ページ。イベント参加レポートのページ。ギャラリーページ。

「anime.html」か「アニメ.html」か。2秒考えて、日本語にした。理由は単純だ。ファイルマネージャーで見たとき、日本語の方が一目でわかる。anime.htmlだと「何のファイルだっけ」と一瞬迷うかもしれない。アニメ.htmlなら迷わない。完璧な判断だと思った。

イベント.html。ギャラリー.html。プロフィール.html。お問い合わせ.html。全部日本語にした。フォルダの中は美しかった。エクスプローラーで見ると、どのファイルが何なのか一目瞭然。整理整頓された本棚のようだった。

そして、この「美しい本棚」が、ひとつずつ小さなカオスを引き起こし始めた。

最初の異変：404エラー

サイトをデプロイして1週間。友人の松本から連絡が来た。

「アニメのページ見ようとしたら404になるんだけど」

「え？ こっちでは見れるよ」

松本のスクリーンショットを見た。確かに404。URLを見ると、「アニメ」の部分が文字化けしている。松本が使っていたのは少し古いバージョンのブラウザだった。日本語のファイル名を正しくエンコードできず、リクエストが壊れていたのだ。

最新のChromeでは問題なく動く。でも、Firefoxの特定のバージョンで挙動が違う。古いSafariでも問題が出る。自分のメインブラウザでは動くから気づかなかった。またこのパターンだ。「自分の環境で動いている」は「全員の環境で動いている」ではない。

「このURL怪しくない？」

2つ目の問題はもっと厄介だった。サイトのURLをTwitterに貼ったら、フォロワーからこんなリプライが来た。

「このURL、なんかフィッシングサイトっぽくない？ %E3%とかいっぱい並んでて怖い」

心臓が止まるかと思った。確認した。Twitterに貼られたURLはこうなっていた。

https://example.com/%E3%82%A4%E3%83%99%E3%83%B3%E3%83%88.html

「イベント.html」のURLだ。自分で見ても確かに怪しい。パーセントエンコーディングの文字列が延々と並んでいる。日本語のファイル名は、URLに変換された瞬間に「読めない文字列」になる。そして読めない文字列は、人に不安を与える。

これが「event.html」だったら、URLはhttps://example.com/event.htmlだ。シンプル。クリーン。誰も怪しいと思わない。「わかりやすさ」のために日本語にしたはずが、URLの世界では逆に「わかりにくさ」を生んでいた。

git地獄

3つ目の問題はgitだった。ターミナルでgit statusを打った。日本語のファイル名が文字化けしている。

modified: "\343\202\242\343\203\213\343\203\241.html"

読めない。どのファイルが変更されたのかわからない。git diffを打っても同じだ。コミットメッセージに「アニメ.htmlを修正」と書いても、diff上ではどのファイルなのか直感的にわからない。

git config core.quotepath falseを設定すれば日本語表示にはなる。でも、これは自分のローカル環境の設定だ。他の人が僕のリポジトリをcloneしたら、また文字化けする。

実際、GitHubでリポジトリを公開したとき、海外のデベロッパーからissueが来た。英語で書かれたissueだった。

「I can't read the filenames. What is this encoding?」

ファイル名が読めない、と。当然だ。日本語が読めない人にとって、「アニメ.html」は意味不明な文字の羅列でしかない。日本語がわかる人にとっての「わかりやすさ」は、日本語がわからない人にとっての「読めなさ」だった。

「Sorry, I'll rename all files to English」と返信した。申し訳なさと恥ずかしさで、キーボードを打つ指が熱かった。

30分のリネーム作業

ファイル名を全部英語に変えることにした。作業リストを作った。

アニメ.html → anime.html。イベント.html → event.html。ギャラリー.html → gallery.html。プロフィール.html → profile.html。お問い合わせ.html → contact.html。他にも8ファイル。合計13ファイルのリネーム。

ファイル名を変えるだけなら一瞬だ。でも、ファイル名を変えるということは、そのファイルへのリンクを全部変えるということだ。ナビゲーションの中のhref。他のページからの内部リンク。CSSのbackground-imageで参照しているパス。JavaScriptの中のURL文字列。全部grep検索して、全部修正した。

13ファイルのリネームと、関連する42箇所のリンク修正。所要時間30分。最初から英語にしていれば、この30分は存在しなかった。

友人の田中に「ファイル名全部英語に直した」と報告した。

「最初から英語にすればよかったのに」

「……わかってる」

「日本語ファイル名にする人、初めて見た」

「……そんなに珍しいの？」

「うん。普通は最初からやらない」

知らなかった。Web開発では日本語ファイル名を避けるのが「普通」らしい。でも、誰も教えてくれなかった。どの入門チュートリアルにも「ファイル名は半角英数字にしましょう」とは書いてあった気がする。読み飛ばしていた。あの1行がこんなに重要だったとは。

「自分にわかりやすい」の落とし穴

この経験で学んだことがある。「自分にわかりやすい」と「みんなにわかりやすい」は違う。

エクスプローラーの中では日本語ファイル名は最高だった。一目で内容がわかる。でもそのファイルがWebに公開された瞬間、URLという別の文脈に置かれる。gitに追加された瞬間、ターミナルという別の文脈に置かれる。GitHubに公開された瞬間、国際的な文脈に置かれる。ファイル名はファイルマネージャーの中だけで生きているわけじゃない。

日本語が悪いわけじゃない。日本語は美しい言語だ。でも、使う場所がある。HTMLファイルのファイル名は、その「場所」ではなかった。コードのコメントに日本語を書くのはいい。READMEに日本語で説明を書くのもいい。でもファイル名は、コードとURLとターミナルとGitHubの交差点にいる。その交差点では、ASCII文字が共通言語だ。

リネーム後、エクスプローラーでファイル一覧を見た。anime.html、contact.html、event.html、gallery.html、profile.html。英語だ。一目で内容がわかるかと言われると、日本語のときほどの即時性はない。でも、URLはクリーンになった。gitは正しく動く。世界中の誰でも読める。

小さな不便を1つ受け入れて、大きな不便を5つ消した。命名規則は最初に決めるべきだった。

命名規則は最初に決めるべき。日本語が悪いんじゃなくて、使う場所を間違えただけだ。
